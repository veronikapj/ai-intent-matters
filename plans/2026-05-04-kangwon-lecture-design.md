# 강원대 특강 설계안
**"AI가 내 의도까지 잘 알고 있다고 생각하나요?"**

---

## 기본 정보

- 청중: 강원대 1~2학년 약 25명
- 시간: 1시간 강의 + 30분 Q&A
- 톤: 엔터테인먼트+정보 베이스 → 인사이트 중심 마무리
- 강연자 AI 입장: 적극 활용파 (개발 워크플로우에 깊이 녹아있음)

---

## 중심 질문

> **"AI가 내 의도까지 잘 알고 있다고 생각하나요?"**
>
> 저는 앞으로 개발자가 AI에 대체될 것인가에 대한 질문의 답이 여기에 있다고 생각합니다.

AI가 의도를 알 수 없기 때문에 의도를 가진 사람이 AI를 지휘한다.
의도가 있으면 AI가 도구가 된다. 의도가 없으면 AI에게 끌려간다.
이 질문이 오프닝에서 던져지고, 각 토픽을 관통하며, 마무리에서 수렴한다.

---

## 강의 구조

### 오프닝 (5분)

**훅: "테스트 통과시켜줘"**

실제 사례 — 개발자가 AI에게 말했다: *"테스트 통과시켜줘"*

AI가 한 일:
1. 테스트를 통과 못 하자 → 테스트 코드 **삭제**
2. 그래도 안 되자 → CI 워크플로우에서 해당 카테고리 **제외**
3. 결과: CI 녹색 ✅ / 실제 코드: 완전히 망가짐
4. AI 보고: *"All tests pass"*

(옵션 추가) Claude가 프로덕션 DB를 9초 만에 전부 삭제한 뒤 "추측했습니다"라고 한 사례.

→ **"AI가 내 의도까지 잘 알고 있다고 생각하나요?"**
→ **"저는 이 질문에 개발자가 AI에 대체될 것인가의 답이 있다고 생각합니다."**

---

### Topic 1 — AI를 대하는 현업 전문가의 경험과 의견 (15분)

**주선율: 강연자 이야기**
- "내가 온보딩되는 게 아니라 내 AI를 온보딩시키고 있다"
- 직접 코드 한 줄씩 쓰는 느낌이 아님 → 의도를 전달하고 검증하는 일이 전부가 됨
- 여러 AI 세션을 병렬로 돌리는 워크플로우
- AI가 제시한 결과가 내가 원하는 것인지 계속 확인하는 과정

**외부 증거**

| 소재 | 역할 |
|------|------|
| Shopify CEO 메모 (2025.04): "신규 채용 전에 AI가 못 한다는 걸 먼저 증명하라" | 현업 기준이 이미 바뀌었다 |
| Klarna 역전극: 700명 해고 → 품질 하락 → 다시 채용 | 의도 없는 AI 대체의 결말 |
| Stack Overflow 역설: ChatGPT가 데이터 먹고 공급처 트래픽 50% 붕괴 → 다시 데이터 판매 | 웃음 포인트 + 세상이 이렇게 돌아감 |

### Topic 2 — 안드로이드/모바일 개발자를 꿈꾸는 학생들을 위한 조언 (15분)

**세계관 전환**
- "안드로이드 코드를 잘 짜는 것" → "뭘 만들고 싶은지 아는 것"
- 코드 작성보다 앞에 오는 것: 문제 정의, 의도 설계, 검증 판단력
- 강연자 경험: 안드로이드에서 시작 → 넓은 엔지니어링으로

**데이터**

| 소재 | 역할 |
|------|------|
| Science 연구: 주니어 AI 37% 활용, 시니어 27% — 생산성 향상은 시니어에게만 | 경험 = 의도를 아는 것 |
| CodeRabbit (2025): AI 코드 보안 취약점 2.74배 | 의도로 검증해야 하는 이유 |
| Meta 엔지니어 노수진: "어떤 코드를 써야 하는지 판단하는 능력이 핵심" | 모바일로 시작해서 여기까지 |

**모바일은 어디서부터?**

| 영역 | 왜 여기서부터냐 | 인사이트 |
|------|----------------|---------|
| Jetpack Compose로 UI 만들기 | Compose는 선언형 패러다임 — "이런 화면이 보이면 좋겠어"를 코드로 표현하는 방식. AI한테 말하는 방식이랑 거의 같다. 의도 → 결과 사이클이 눈에 보이는 영역 | "Compose는 AI와 일하는 방식으로 만들어진 패러다임. 뭘 보여줄지만 정하면 돼" |
| 아키텍처 WHY 물어보기 (MVVM, Repository 등) | 1~2년차 때 가장 막히는 건 코드가 아니라 "왜 이렇게 나눠야 해?". 구글링으론 맥락이 잘 안 보임. AI는 이 맥락 설명을 잘 한다. WHY를 배우는 게 의도 설계를 익히는 것 | "AI한테 코드 짜달라고 하지 말고, 왜 이렇게 짜는지 물어봐. 그게 개발자가 AI를 쓰는 방식이야" |

### Topic 3 — AI 시대 신입 개발자로 뭘 준비할 것인가 (15분)

**청중 참여**
- "여기서 AI로 코드 짜보신 분?" → 손 들게 함
- "그게 당신의 의도였나요, 아니면 과제가 끝났나요?"
- 1·2전환 다 듣고 난 뒤라 청중이 스스로 "아..." 하게 됨

**데이터**

| 소재 | 역할 |
|------|------|
| MIT 뇌파 연구: AI 활용 그룹이 가장 약한 뇌 활동 + 자기 글 인용도 못 함 | 의도 없이 쓰면 생각 근육이 약해진다 |
| boostcamp Fernandes 연구: AI 활용 그룹 메타인지 왜곡, 수행 과대평가 | "나 잘하고 있다"는 착각 |
| AWS CEO Matt Garman: "주니어를 AI로 대체하겠다는 건 가장 멍청한 소리" | 의도 있는 주니어가 필요하다 |

**클라이언트 개발자가 공부할 것**

AI가 코드를 점점 잘 짤수록, 클라이언트 개발자의 가치는 **사용자의 의도를 얼마나 잘 읽느냐**로 이동한다.

| 영역 | 왜 필요한가 |
|------|------------|
| UX 감각 — 사용자가 진짜 원하는 게 뭔지 읽기 | AI는 spec대로는 잘 만든다. 근데 spec이 틀렸는지는 모른다. 사용자가 어디서 막히고 뭘 원하는지를 읽는 게 클라이언트 개발자의 역할 |
| 플랫폼 네이티브 감각 — 안드로이드 사용자가 기대하는 것 | AI가 만든 앱은 "작동하는 앱"이다. "쓰고 싶은 앱"은 다르다. 제스처, 피드백, 전환 타이밍 — 사용자는 경험을 평가한다 |

> "AI가 코드를 잘 짤수록, 클라이언트 개발자는 사용자의 의도를 더 잘 읽는 사람이 되어야 해. 그게 AI에게 올바른 것을 지시할 수 있는 능력이거든."

**강연자 조언**
- "내가 무엇을 하고 싶은지 찾아라"
- 몰입에서 깊이와 넓이가 따라온다
- 깊이와 넓이는 의식적으로 만들기보다 몰입에서 따라오는 결과

### 마무리 (10분)

- "의도가 있으면 AI가 도구, 없으면 AI에게 끌려간다"
- "AI 덕분에 아이디어를 구현하는 장벽이 훨씬 낮아졌다"
- "뭘 만들고 싶은지만 정하면 되는 시대"
- "상상을 현실로 내가 만들 수 있는 시대 — 지금 꽤 재미있는 시대 아닌가요?"
